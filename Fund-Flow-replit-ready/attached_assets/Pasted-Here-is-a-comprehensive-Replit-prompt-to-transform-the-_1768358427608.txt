Here is a comprehensive Replit prompt to transform the **Driver Sensitivity Analysis** screen into a more interactive, informative, and unique feature for founders. The prompt outlines both front‑end and back‑end tasks to achieve the UX improvements suggested earlier.

**Replit Prompt – Enhance Driver Sensitivity Analysis for Founders**

I am working on the **Sensitivity** tab of our Scenario Builder in a React/Next.js application. This tab currently shows a “Driver Sensitivity Analysis” with a basic list of key drivers (e.g., growth rate, churn rate) and displays targets to achieve a specific runway. However, the UX is minimal (e.g., showing “0 % impact”) and not very actionable. I want to overhaul this screen to make it an interactive dashboard that guides founders on which levers to pull to reach their runway and probability targets. Please implement the following improvements:

### 1. Make Targets User‑Defined and Clearly Explained

* In the “What Must Be True” header, add editable fields for:

  * **Target Runway (months)**: default to 18 months but allow the user to input any number (e.g., 12–36).
  * **Target Probability (%)**: default to 70 % but user can set between 50–100 %.
* Show a tooltip or helper text explaining these fields (e.g., “Your goal is to have cash in the bank for X months with Y % confidence under your current assumptions.”).
* When the user updates these targets and clicks a “Run Analysis” button, recompute the sensitivity results and refresh the page.

### 2. Visualize Current vs. Target Metrics

* For each key driver, create a card or row that displays:

  * **Driver name and icon** (e.g., rocket for growth, arrow down for churn).
  * A description of how the driver affects runway (“Decreasing churn rate improves runway…”).
  * **Current value vs. Target threshold** (e.g., “Current: 3 % / Target: 2.1 %”).
  * A **progress bar or gauge** showing how close the current value is to the target:

    * If the driver needs to increase (e.g., growth rate), the bar fills from left to right, coloured red/yellow/green depending on proximity.
    * If the driver needs to decrease (e.g., churn rate), the bar fills right to left.
* If the current value meets or exceeds the target threshold, display a green checkmark or “Achieved” tag; otherwise, show how far away it is in red (“–0.9 %”).

### 3. Calculate and Display Driver Impact Percentages

* Update the sensitivity analysis algorithm to calculate the **impact percentage** for each driver. This is the contribution of a small change in the driver to the probability of reaching the target runway.

  * Example: A 1‑point reduction in churn might increase survival probability by 5 %, while a similar change in growth rate might contribute 10 %. Normalize these impacts so the top driver shows the highest percentage.
* In the UI, replace the current “0 % impact” labels with actual values (e.g., “Growth Rate: 25 % impact”).
* Sort drivers by impact in descending order by default, with an option to sort by name or by “distance to target”.

### 4. Provide Actionable Recommendations

* For each driver that is not currently within the target threshold, generate a suggestion:

  * Determine the minimum change needed to close the probability gap. For example, “Increase growth rate by +4 percentage points to reach 70 % survival.”
  * If the recommended change is unrealistic (e.g., growth rate must jump from 10 % to 50 %), display a caution (“This may be hard to achieve based on historical performance”).
* Display these recommendations inline with the driver card or in a dedicated “Recommendations” column.

### 5. Break Down the Probability Gap

* In the “What Must Be True” section, currently the Probability Gap is displayed (e.g., 30 %). Break this down by showing how much each driver contributes to closing the gap.

  * Example: “Probability Gap: 30 % (Growth Rate: 18 %, Churn Rate: 7 %, Pricing: 5 %)”.
* Represent this breakdown visually (e.g., stacked bar or pie chart) to illustrate where the biggest opportunities lie.

### 6. Interactivity and Scenario Integration

* For each driver, provide a “What if?” button. When clicked:

  * Take the user back to the main Scenario Builder with a pre‑filled parameter adjustment (e.g., increase growth uplift by 3 %, decrease churn by 1 %), so they can run a new simulation and immediately see the effect on runway.
  * Use query parameters or state to pass these adjustments through the router without requiring manual input.

### 7. Optional Benchmarking

* If data is available, compare the company’s metrics with **industry benchmarks**. For example:

  * Show that the average churn rate for similar SaaS startups is 2.5 %, and the company is currently at 3 %.
  * Display a small comparison bar or gauge showing the benchmark value and highlight if the target threshold is above/below typical performance.
* This can help founders see whether their targets are realistic or overly ambitious.

### 8. Improved Layout & Aesthetics

* Use a two‑column layout:

  * Left column: “What Must Be True” summary with target runway, target probability, current probability, probability gap breakdown.
  * Right column: a scrollable list of driver cards.
* Use consistent card sizes and spacing. Use colour coding (green/yellow/red) to indicate progress toward targets.
* Ensure that all text is legible, with enough contrast for dark/light themes.

### 9. Back‑End Updates

* Extend the sensitivity analysis endpoint (`/api/scenario/{id}/sensitivity`) to:

  * Accept target runway and probability parameters.
  * Return, for each driver:

    ```json
    {
      "name": "growth_rate",
      "currentValue": 0.12,
      "targetThreshold": 0.16,
      "impact": 0.25,
      "recommendedChange": 0.04,
      "achievable": true
    }
    ```
  * Return an array of drivers sorted by impact, and a breakdown of the probability gap by driver.
* If benchmark data is available, include it in the response.

### 10. Testing & Documentation

* Write unit tests for:

  * Correct calculation of driver impacts and recommended changes.
  * Handling of user‑defined targets and recomputing results.
  * Display of progress bars and colour status based on current vs. target values.
* Update documentation to describe the new API parameters and the user‑facing features (editable targets, progress visualization, recommendations, benchmarking).

---

**Outcome**

After implementing these changes, the Sensitivity tab will transform from a static list into an interactive dashboard. Founders will understand at a glance which drivers matter most, how close they are to achieving their goals, and what specific adjustments are needed to close the probability gap. The ability to tweak targets, visualize progress and benchmark against peers will make this feature a compelling differentiator and a unique selling point for your financial simulator.
