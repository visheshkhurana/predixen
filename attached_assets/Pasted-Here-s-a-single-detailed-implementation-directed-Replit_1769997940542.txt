1) Canonical Run Results Contract (Single Source of Truth)

1.1 Create/confirm canonical endpoint:

GET /api/runs/:runId

Returns RunResult with provenance, metrics, status, and validation flags.

Response shape (RunResult):

type RunResult = {
  companyId: string;
  scenarioId: string;
  scenarioName: string;
  runId: string;
  runTimestamp: string;        // ISO
  dataSnapshotId: string;      // hash of company_state
  status: "queued" | "running" | "completed" | "failed" | "invalid";
  metrics?: {
    runwayMonths: { p10: number; p50: number; p90: number };
    survival18mo: number;      // 0..1
    cashBalance: number;
    netBurn: number;
    cashEnd?: { p10: number; p50: number; p90: number };
  };
  validation?: {
    runwayCashBurnMismatch?: boolean;
    survivalRunwayMismatch?: boolean;
    monteCarloZeroVariance?: boolean;
    notes?: string[];
  };
};

1.2 Endpoint to find latest run by scenario:

GET /api/company/:companyId/scenarios/:scenarioId/runs/latest

Returns:
	•	latest completed run (preferred)
	•	else latest failed/invalid with error (so UI can show failure)

5) Strict Output Modes (COP-013)

Implement schema-locked response envelopes.

5.1 Unified response envelope from Copilot backend

type CopilotEnvelope =
  | { kind: "NARRATIVE"; text: string; provenance?: ProvenanceBlock }
  | { kind: "JSON"; json: any; provenance?: ProvenanceBlock }
  | { kind: "TOKEN"; token: "OK" | "NOT_AVAILABLE" | "UNVERIFIED_NO_RUN" | "UNVERIFIED_MISMATCH" | "MISSING_INPUTS" | "UNKNOWN_SURVIVAL_DEFINITION" };

5.2 Request parsing

If user prompt includes patterns:
	•	“JSON only” / “respond in JSON” → kind=JSON
	•	“one token only” → kind=TOKEN
	•	If UI is performing automation checks → call deterministic backend, not LLM.

5.3 Enforce at server
	•	Validate shape before returning to UI
	•	If invalid, auto-retry once with a stricter system prompt
	•	If still invalid, return:

{ kind: "TOKEN", token: "NOT_AVAILABLE" }
