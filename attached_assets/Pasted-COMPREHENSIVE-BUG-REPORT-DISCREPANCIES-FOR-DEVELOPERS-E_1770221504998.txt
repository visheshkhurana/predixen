COMPREHENSIVE BUG REPORT & DISCREPANCIES FOR DEVELOPERS
Executive Summary
During comprehensive testing of the Predixen application with the CloudScale Analytics demo company, several discrepancies and bugs were identified between data entered in the Data Input module and values calculated/displayed in the Truth Scan module. This report documents all issues with reproduction steps and expected behavior.

CRITICAL ISSUES
ISSUE #1: Revenue Data Mismatch Between Modules
Severity: 🔴 HIGH
Category: Data Consistency / Calculation Error
Status: OPEN
Problem Description
Data entered in the Data Input > Financials tab does not match the revenue displayed in the Truth Scan > Key Metrics section.
Reproduction Steps

Navigate to Data Input > Financials tab
Enter Monthly Revenue: $82,500
Click "Save All Data"
Navigate to Truth Scan
Observe "Monthly Revenue" in Key Metrics section

Expected Behavior
Truth Scan should display: $82,500/month
Actual Behavior
Truth Scan displays: $75,000/month
Data Variance
Entered Value:      $82,500
Displayed Value:    $75,000
Discrepancy:        $7,500 (9.1% difference)
Root Cause Analysis Required

Is the Truth Scan recalculating from historical data?
Is there a different data source being used?
Is there rounding/normalization happening?
Is the API call returning different data than submitted?

Affected Calculations
This discrepancy cascades to:

MRR (shows $75K instead of $82.5K)
ARR (shows $900K instead of $990K)
Net Burn calculations
Runway projections
Unit economics (ARPU, LTV, CAC ratios)

Development Notes

Check calculateMRR() function
Review data synchronization between /data endpoint and /truth endpoint
Verify calculation formula in Truth Scan module
Check if historical average is being used instead of current month


ISSUE #2: Total Monthly Expenses Mismatch
Severity: 🔴 HIGH
Category: Data Consistency / Missing Logic
Status: OPEN
Problem Description
Total Monthly Expenses entered in Data Input form does not match the sum of expenses calculated and displayed in Truth Scan's "Monthly Burn Breakdown".
Reproduction Steps

Navigate to Data Input > Financials tab
Enter in Expense Breakdown:

Payroll: $41,000
Marketing: $10,200
Operating: $11,000
COGS: $16,500
Other: $0


System auto-calculates Total Monthly Expenses: $78,700
Click "Save All Data"
Navigate to Truth Scan > Key Metrics
Scroll to "Monthly Burn Breakdown"

Expected Behavior
Monthly Burn Breakdown should total: $78,700 matching the entered values
Actual Behavior
Monthly Burn Breakdown displays: $93,000 with additional line items:
Payroll:      $45.0K (48%)
G&A:          $15.5K (17%)     [NOT ENTERED]
COGS:         $15.0K (16%)
Marketing:    $10.0K (11%)     [Discrepancy: entered $10.2K]
R&D:          $7.5K  (8%)      [NOT ENTERED]
────────────────────────────
Total:        $93.0K
Data Variance Summary
Entered Total:         $78,700
Displayed Total:       $93,000
Difference:            $14,300 (18.1% increase)

New/Modified Categories in Truth Scan:
├── G&A (added):       +$15,500
├── R&D (added):       +$7,500
├── Marketing (mod):    $10,200 → $10,000 (-$200)
└── Payroll (mod):     $41,000 → $45,000 (+$4,000)
Root Cause Analysis Required

Where is the G&A expense of $15,500 derived from?
Where is the R&D expense of $7,500 derived from?
Is the system applying expense ratios based on industry benchmarks?
Are these derived from revenue percentages?
Is there automatic allocation logic that isn't documented?

Impact

Burn rate calculations are inflated by $14,300/month
Runway projections are pessimistic
Profitability assessment is skewed
User sees different numbers than they entered

Development Notes

Search for expense allocation or benchmarking logic
Check calculateBurnBreakdown() function
Review if G&A and R&D are being automatically added
Verify formula in expense category normalization
Document any automatic expense category additions

CONCLUSION
The Predixen application demonstrates strong core functionality with accurate calculations in the Data Input form. However, there are significant data consistency issues between modules that must be resolved before launch. The three critical issues (Issues #1, #2, #3) affect financial accuracy and user trust in the system.
Overall Assessment:

✅ Data input mechanism works well
❌ Data consistency between modules has gaps
⚠️ Some automatic calculations are unexplained
🔄 File upload feature needs completion

Recommended: Schedule architecture review of data flow between modules and implement comprehensive testing for all calculation paths.
