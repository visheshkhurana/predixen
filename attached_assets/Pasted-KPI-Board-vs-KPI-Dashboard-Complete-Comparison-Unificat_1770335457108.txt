KPI Board vs. KPI Dashboard: Complete Comparison & Unification Strategy
The Three Views (There Are Actually Three, Not Two)
After deep testing, Predixen actually has three separate metric-display surfaces, each pulling from a different backend API, sitting under different navigation sections, and serving different purposes. This is the root of the fragmentation.

1. KPI Board (Analytics → KPI Board, /kpi-board)
Data Source: GET /api/realtime/kpi/74/snapshot — a dedicated real-time snapshot endpoint that returns pre-computed metrics directly. Status: 200 ✅
What it is: A fixed, system-generated dashboard with 11 hardcoded KPI cards (MRR, ARR, Cash Balance, Runway, Net Burn, Gross Margin, Churn Rate, Headcount, CAC, LTV, LTV/CAC Ratio) plus 4 preset trend charts (Revenue Trend, Burn & Runway, Unit Economics, Margins & Churn). It has a "Live" badge and auto-refreshes with a last-update timestamp.
Strengths: Works out of the box, no setup needed. Shows real-time data from whatever has been input. Has trend visualizations.
Weaknesses: Completely rigid — you cannot add, remove, rearrange, or customize any card. No segmentation, no benchmarking, no health scoring. Data flickers to $0 on re-mount. Shows only the most recent snapshot with no historical depth.

2. KPI Dashboards (Stakeholders → KPI Dashboards, /dashboards)
Data Source: GET /api/dashboards/2 for layout + GET /api/metrics?company_id=74 for metric data — depends entirely on the Metric Catalog. Status: 500 ❌
What it is: A fully customizable, widget-based dashboard builder. You name it, describe it, then compose it from four widget types: KPI Card (big number with trend), Line Chart (trend over time), Bar Chart (compare values), and Table (data table). You pick a metric from the Metric Catalog for each widget, give it a custom title, and drag/resize to arrange your layout. You can create multiple dashboards (it's at /dashboards, plural).
Strengths: Complete customization — choose which metrics, what visualization, how they're arranged. Supports multiple dashboards for different audiences (investor board, ops team, board meeting). Placed under "Stakeholders" — clearly designed for external-facing reporting.
Weaknesses: Currently 100% non-functional because it depends on the Metric Catalog API which returns 500 errors. The metric selector dropdown is empty since no metrics exist. Cannot add a single widget until the Metric Catalog is populated. Has no data of its own — it's purely a presentation layer on top of the Metric Catalog.

3. Overview (Analytics → Overview, /)
Data Source: GET /api/companies/74/truth/latest + GET /api/benchmarks/search — pulls from Truth Scan data and industry benchmark data. Status: 200 ✅
What it is: The richest view of all three. Shows MRR, ARR, Cash on Hand with Runway, Burn Rate, CAC, LTV with health labels. Then adds ARPU ($467), Active Users (224), NRR (99.5%), LTV:CAC Ratio (4.5x). Includes Segment Analysis (by Channel, Tier, Region), KPI Health Status with color-coded threshold bars, Quality of Growth Index (75/100), Milestones, Industry Benchmarks with percentile rankings, Notes & Annotations, and an AI Copilot prompt.
Strengths: Most comprehensive intelligence layer. Has benchmarking, segmentation, health scoring, and AI analysis. Functional and populated.
Weaknesses: Also not customizable — fixed layout. Uses different metric values from the KPI Board in some cases (Runway: 3.7 mo vs. 16.1 mo, Churn Rate: 5.5% vs. 2.0%, Headcount implied at a different number). This inconsistency exists because the Overview pulls from /truth/latest while the KPI Board pulls from /realtime/kpi/snapshot — two separate computation pipelines.

The Core Problem: Three APIs, Three Truths
The fragmentation isn't just a UI issue — it's an architectural split at the data layer. The KPI Board snapshot API calculates metrics one way (apparently from the Data Input financials), the Truth Scan API calculates them differently (with cross-validation and adjustments), and the Metric Catalog API (which the Dashboard depends on) is broken entirely. The result is that a founder looking at Runway sees 16.1 months on the KPI Board, 3.7 months on the Overview, and nothing on the Dashboard.

How to Unify
Step 1: Fix the Metric Catalog API (Prerequisite). The 500 errors on POST /api/metrics/initialize and GET /api/metrics?company_id=74 must be resolved by the Predixen team. This is a backend issue — the metrics table or service is likely misconfigured for company_id 74. Report this to support with the specific endpoint details. Until this works, unification is impossible because the Dashboard has no data to display.
Step 2: Initialize System Metrics to create the canonical metric definitions. Once the API is fixed, the ~24 standard SaaS metrics should be auto-created in the Metric Catalog with their formal formulas, units, and grain. This becomes the single source of truth that all three views should eventually reference.
Step 3: Use the KPI Dashboard as your primary external reporting surface. Create audience-specific dashboards: an "Investor Board Update" dashboard with MRR, ARR, Burn Rate, Runway, LTV/CAC, and Growth Rate as KPI cards plus a Revenue Trend line chart; a "Weekly Ops" dashboard with Churn Rate, NRR, Headcount, and CAC as KPI cards with a bar chart comparing expense categories; and a "Board Meeting" dashboard with the full financial picture in table format. The widget-based builder is specifically designed for this — KPI Cards for headline numbers, Line Charts for trends, Bar Charts for comparisons, Tables for detailed breakdowns.
Step 4: Treat the KPI Board as the internal quick-glance monitor. Since it auto-refreshes and works without configuration, it serves as the real-time operational pulse — the view you keep open during the workday. Don't try to customize it; accept its fixed layout as a feature for speed.
Step 5: Treat the Overview as the intelligence and analysis layer. The Overview is where you go for context — benchmarks, health scoring, segment analysis, growth quality. It answers "how am I doing relative to peers" while the KPI Board answers "what are my numbers right now."
Step 6: Push for API convergence from Predixen's side. The long-term fix is for all three views to read from the same Metric Catalog definitions. Ideally, the real-time snapshot API (/realtime/kpi/snapshot) should compute values using Metric Catalog formulas rather than its own independent logic, and the Truth Scan should validate against those same definitions rather than recalculating with different assumptions. Until Predixen unifies the backend computation, you'll continue to see the Runway 16.1 mo vs. 3.7 mo discrepancy — which is dangerous for decision-making.
In short: the KPI Board is your speedometer, the Overview is your diagnostic panel, and the KPI Dashboard (once working) is your customizable report generator for stakeholders. All three have a role, but they need the Metric Catalog as a shared foundation to stop contradicting each other.
