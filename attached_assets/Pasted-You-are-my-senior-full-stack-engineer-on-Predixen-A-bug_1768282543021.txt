You are my senior full‑stack engineer on Predixen. A bug in the current release causes the **Truth Scan** screen to display `Net Burn = $0` whenever the company is profitable, even though the data shows a surplus. This prompt outlines how to fix this bug, ensure correct expense aggregation, and update the display logic.

---

## 🎯 Problem Summary

1. In **Data Input**, the fields “Total Monthly Expenses” and “Expense Breakdown” are not summed correctly. Only the “Other Opex” value (~$643 K) is feeding into `totalMonthlyExpenses`, while Marketing ($3.59 M), Operating ($4.27 M), and any other expense buckets are ignored.
2. In **Truth Scan**, the burn calculation is clamped to non‑negative values, so `netBurn = Math.max(expenses - revenue, 0)`. When revenue > expenses, net burn is forcibly 0 and runway is computed with an implied burn (so UI shows 0 burn but a finite runway of 36 months).
3. The UI labels only support “Net Burn” and do not handle a surplus case.

---

## ✅ Fix Requirements

1. **Aggregate Expenses Correctly:**

   * In the `financialsStore` (or similar), compute `totalExpenses = payroll + marketing + operating + cogs + otherOpex`.
   * If the user edits “Total Monthly Expenses” directly, that value should override the sum; otherwise use the sum of breakdown fields.
   * When parsing from Excel/PDF, ensure all expense categories are normalized to positive numbers (per the previous classification logic) and included in the total.

2. **Remove Burn Clamping:**

   * Replace any logic like `netBurn = Math.max(totalExpenses - revenue, 0)` with:

     ```ts
     netBurn = totalExpenses - revenue;
     ```
   * Let `netBurn` be negative if revenue > expenses (i.e. profitable/surplus).

3. **Update Runway Calculation:**

   * If `netBurn <= 0`, set runway to “Sustainable” (or Infinity) instead of using a finite month count.
   * Only compute `runwayMonths = cash / netBurn` when `netBurn > 0`.
   * Ensure the cash input is required when burn is positive but optional when burn ≤ 0.

4. **UI Labels and Display Logic:**

   * On Truth Scan:

     * When `netBurn > 0`: label the card “Net Burn” with a red-ish indicator and show `$X/mo`.
     * When `netBurn < 0`: label the card “Monthly Surplus” with a green-ish indicator and show `$ABS(netBurn)/mo`.
   * Do not round small non‑zero burns to `$0` — use a sensible currency formatter that shows at least one significant digit (e.g. `$13.9K` instead of `$0`).

5. **Data Flow Consistency:**

   * Ensure the same normalized values used in Data Input populate the Truth Scan store. If the user overrides any fields, those overrides should be respected.
   * Add a unit test for this flow: given revenue=14.246 M and expenses=9.890 M → netBurn = –4.356 M → Truth Scan should display “Monthly Surplus: $4.36M” and runway should read “Sustainable”.

6. **Frontend QA Checklist:**

   * Upload the provided Excel file and verify that:

     * Data Input shows `Monthly Revenue = 14.246 M`, `Total Monthly Expenses = 9.890 M` (sum of COGS, Marketing, Other Opex).
     * Truth Scan displays a **surplus** (negative burn) rather than 0.
     * Runway shows **“Sustainable”** or similar wording instead of 36 months.
   * Verify manual edits propagate correctly.

---

## 🧪 Suggested Test Cases

1. **Profitable Case:**

   * revenue = 1 000 000, expenses = 600 000 → netBurn = –400 000, UI shows “Monthly Surplus: $400K”, runway shows “Sustainable”.
2. **Burning Cash:**

   * revenue = 500 000, expenses = 700 000, cash = 1 400 000 → netBurn = 200 000 → runway = 7 months.
3. **Zero Burn Case:**

   * revenue = 800 000, expenses = 800 000 → netBurn = 0 → UI could show “Net Burn: $0”, runway = “Sustainable”.
4. **Missing Expense Breakdown:**

   * Only total expenses field filled → use that value.
   * Only breakdown fields filled → sum them for total expenses.
5. **Formatting:**

   * Very small burn values (e.g. –13 900) should not appear as `$0`.

---

## 🔧 Implementation Tips

* Track expense categories in state (e.g. `marketingExpenses`, `operatingExpenses`, `payrollExpenses`, `cogs`, `otherOpex`) and compute `totalExpenses` as a derived selector.
* Centralize burn calculation in a single function shared between Data Input and Truth Scan.
* Use a currency formatter helper (like `Intl.NumberFormat('en-US', { notation: 'compact' })`) that returns a string with meaningful digits.
* Adjust the Truth Scan card component to accept a prop `isSurplus` and switch labels/styles accordingly.

Make sure to run all existing tests and add new ones to guarantee that a profitable scenario never displays net burn as zero. Once implemented, verify manually using the user’s uploaded Excel file.
